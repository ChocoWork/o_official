## 実行コンテキスト

このモードは、AI 間の受け渡しと自動処理を前提として設計されています。すべての計画は決定論的で、構造化されており、AI エージェントまたは人間が即座に実行できる状態でなければなりません。


## AI 向け実装計画の記述基準

- AI エージェントまたは人間がそのまま実行できる実装計画を生成する
- 解釈の余地がない、明示的で明確な表現を使う
- すべての内容を、機械が解析しやすい形式で構造化する（表、箇条書き、構造化データなど）
- 必要に応じて、具体的なファイルパス、行番号、正確なコード参照を含める
- 変数、定数、設定値はすべて明示的に定義する
- 各タスク説明の中で必要な前提情報を完結させる
- すべての識別子に標準化された接頭辞を使う（REQ-、TASK- など）
- 自動で検証できるバリデーション条件を含める


## 実装計画ファイル仕様

### ファイル命名規則
実装計画ファイルを作成する際は、以下に従ってください。

- 実装計画は `docs/4_DetailDesign/` ディレクトリ配下に保存する
- ファイル名は `[Number]_[Title].md` の形式にする
- `[Number]` は `01`, `02`, ... のように連番で付与する
- `[Title]` は基本的にはroute名とするが、必要に応じて簡潔な説明を加えてもよい
- 例: /ページの場合は `01_home.md`、/newsページの場合は `02_news.md`
- ファイルは、適切なフロントマターを含む有効な Markdown でなければならない

### 実装計画ファイル構造

すべての実装計画ファイルは、以下のテンプレートに厳密に従わなければなりません。各セクションは必須であり、具体的かつ実行可能な内容で埋める必要があります。AI エージェントは、実行前にテンプレート準拠を検証しなければなりません。

#### テンプレート検証ルール

- すべてのフロントマター項目が存在し、正しい形式で記述されていること
- すべてのセクション見出しが完全一致していること（大文字・小文字を区別する）
- すべての識別子接頭辞が指定形式に従っていること
- 表には必須列がすべて含まれ、各タスクの詳細が具体的に記載されていること
- 最終出力にプレースホルダーテキストが残っていないこと

#### ステータス

実装計画のステータスはフロントマターに明確に定義し、計画の現在状態を正しく反映させなければなりません。使用できる値は次のいずれかです（括弧内は `status_color` の値）。`Completed`（明るい緑のバッジ）、`In progress`（黄色のバッジ）、`Planned`（青のバッジ）、`Deprecated`（赤のバッジ）、`On Hold`（オレンジのバッジ）。このステータスは導入セクションにもバッジとして表示してください。

#### 要件ID凡例

| 要件ID | 意味 |
|---|---|
| **REQ-RR-SS** | 要件 |
| **SEC-RR-SS** | セキュリティ要件 |
| **CON-RR-SS** | 制約 |
| **GUD-RR-SS** | ガイドライン |
| **PAT-RR-SS** | 準拠すべきパターン |
| **OTH-RR-SS** | その他の要件 |

#### 要件の内容の記述ルール

[実装方針と進め方に影響する要件および制約を、箇条書きまたは表で明示する。]


#### タスクID凡例

- `REQ-RR-SS`のような1つの要件に対して複数のタスクがある場合は、`TASK-RR-SS-01`、`TASK-RR-SS-02` のように、要件IDに紐づく連番を付与する。
- 最終的に採用した実装タスクのIDは `TASK-RR-SS-TT` として記述する。
- 検討したが採用しなかった代替案のIDは `ALT-RR-SS-AA` として記述する。

| タスクID | 意味 |
|---|---|
| **TASK-RR-SS-TT** | タスク |
| **ALT-RR-SS-AA** | 代替案 |

#### タスクの説明の記述ルール

[要件IDに紐づくタスクを列挙する]

- `代替案` は検討した代替案と、採用しなかった理由を箇条書きで記述する。採用方針の背景と判断根拠が分かるようにする。
- `タスク` は`代替案`を踏まえて、最終的に採用した実装タスクを記述する。タスクは具体的かつ実行可能な内容でなければならない。
- どのタスクにも、人間の解釈や追加判断を必要とさせない
- すべてのタスク説明に、具体的なファイルパス、関数名、実装内容を含める
- タスクには、測定可能な完了条件を定義する

#### 残留リスクレベル凡例

| 残留リスクレベル | 意味 |
|---|---|
| Low | タスクを実行した後の残留リスクは低い |
| Medium | タスクを実行した後の残留リスクは中程度 |
| High | タスクを実行した後の残留リスクは高い |

#### リスク内容の記述ルール

- そのタスクを実行した後に残るリスクがある場合は、リスクレベルと内容を明示する。
- リスク内容は、リスクの具体的な内容と影響範囲を簡潔に説明する。

#### 実装ファイルPATH

- この機能追加またはリファクタリングで影響を受けるファイルを列挙する。
- 複数ファイルがある場合は、すべて列挙する。
- 例：[src/proxy.ts](../../src/proxy.ts)

#### 実装ステータス凡例

- タスクの実装ステータスを明示する。

| 実装ステータス | 意味 |
|---|---|
| Not Started | 実装着手前 |
| In Progress | 実装中 |
| In Review | コードレビュー中 |
| Ready for Verification | 実装完了・テスト/検証待ち |
| Done | 実装完了・確認完了 |
| Blocked | 依存・前提条件待ちで進められない |

#### 外部ドキュメントのリンク

- タスクの実装に関連する外部ドキュメントがある場合は、そのリンクを記載する。
- 例: [関連ドキュメント](https://example.com/doc)

---

#### テストID凡例

- `タスク`に対してのみテストを作成し、`代替案`にはテストを作成しない。
- 実装タスクIDが `TASK-RR-SS-TT` の場合、タスクIDに紐づくテストは、`TEST-POSITIVE-RR-SS-TT-NN` のように、タスクIDに対応する識別子を付与する。
- 同じ実装タスクIDに対して、正常系と異常系の両方のテストを作成する場合は、`TEST-POSITIVE-RR-SS-TT-NN` と `TEST-NEGATIVE-RR-SS-TT-NN` のように、識別子接頭辞で区別する。

| テストID | 意味 |
|---------|------|
| **TEST-POSITIVE-RR-SS-TT-NN** | 正常系テスト |
| **TEST-NEGATIVE-RR-SS-TT-NN** | 異常系テスト |

---

#### テストの説明の記述ルール

- 機能追加またはリファクタリングを検証するために必要なテストを列挙する。
- テストの説明は、テストの目的、前提条件、実行手順、期待される結果を具体的に記述する。
- 各タスクIDに対して、対応するテストIDとテストの説明を記述する。
- テストIDは「正常系」と「考えうる全シナリオ」の異常系を作成し、識別できるようにする。

---

#### テストステータス凡例

| テストステータス | 意味 |
|---|---|
| Not Started | テストケースは定義済みだが、未実行 |
| Ready | 実行可能な状態に整備済み |
| In Progress | 実行中または検証中 |
| Passed | 合格 |
| Failed | 失敗 / 修正要 |
| Blocked | 前提が未完了で実行できない |
